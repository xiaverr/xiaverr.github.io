1. 概述与架构

1.1 什么是 LMCache

LMCache 是一个 LLM 推理引擎的 KV 缓存管理扩展，通过跨数据中心存储层 (GPU、CPU、磁盘、远程) 缓存和复用 KV Cache，从而：

  • 降低 TTFT (Time-To-First-Token)：缓存 prefill 结果，避免重复计算相同的 prompt 前缀
  • 提高吞吐量：跨请求共享 KV Cache，减少 GPU prefill 计算量
  • 支持跨实例共享：多个 vLLM 实例之间通过远程后端共享 KV 缓存
  • 支持 Disaggregated Prefill：Prefill/Decode 分离架构下的 KV 缓存传输

8.7 场景 7: KV 混合 (Blending)

多文档拼接场景，LMCache 可以重用各个文档的缓存，混合后仅需少量重计算。

工作流程：

  1. 用户查询涉及多个文档片段 → 拼接为单个长 prompt
  2. LMCache 查找每个文档的缓存 chunk
  3. 在拼接点附近的少量层进行 KV 混合计算
  4. 大部分层直接复用缓存 → 显著减少 prefill 计算

8.8 场景 8: Layerwise 模式 (低内存)

适合 GPU 显存或 CPU 内存有限的场景。

与普通模式的区别：

  • 普通模式: 所有层的 KV 同时保存在 CPU 内存中
  • Layerwise 模式: 逐层 store/retrieve，峰值内存降低 ~N 倍 (N = 层数)

8.9 场景 9: 延迟内存分配 (大缓存场景)

当 max_local_cpu_size 设置较大时，使用延迟分配减少启动内存占用。

11.3 性能调优

  1. chunk_size: 对于长 prompt 场景，增大 chunk_size (512/1024) 减少 key 数量和 hash 计算开销
  2. cache_policy: 对于重复 prompt 场景用 LRU，热点数据场景用 LFU
  3. layerwise: 低内存场景下启用，但会增加延迟 (逐层 I/O 开销)
  4. save_decode_cache: 启用以缓存 decode 阶段 KV，提高多轮对话性能
  5. remote_serde: cachegen 可节省 5-10x 远程传输带宽，但有少量精度损失
  6. GDS: 有 NVMe 设备时优先使用 GDS，避免 CPU 拷贝开销
  7. numa_mode: 多 NUMA 节点系统上配置以优化内存访问
