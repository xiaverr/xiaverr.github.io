【mooncake】mooncake架构分析和代码解读

xiaver2026-04-192026-04-19

什么是mooncake

以 KVCache 为中心的分解架构，将预填充和解码集群分开。利用GPU集群中未充分利用的CPU、DRAM和SSD资源来实现KVCache的分解缓存。它在最大化整体有效吞吐量的同时满足与延迟相关的服务级别目标。为了解决高度超载的场景，mooncake制定了基于预测的早期拒绝政策。

简单来说，mooncake是实现了解码器大模型PD分离。同时为了提高系统吞吐量，充分利用了集群中的DRAM和SSD资源，并且优化了KVCache缓存系统。mooncake还基于预测的早期拒绝政策进一步提高吞吐量。

mooncake架构

Mooncake 采用了一种存算分离架构，不仅将预填充节点与解码节点分离，还通过整合 GPU 集群的 CPU、DRAM、SSD 和 RDMA 资源来实现分离式的 KVCache。

系统中，所有Prefill Pool和Decoding Pool会共享KV Cache。这样做就可以充分利用各种资源了。

所以系统存在三种pool，分别为Prefill Pool和Decoding Pool和KV Cache Pool。这些都是分布式的，所以肯定需要一种中心化的管理机构来进行资源调度。为了调度所有这些解耦组件，Mooncake 在其中心实现了一个名为 Conductor 的全局调度器。 Conductor 负责根据 KVCache 和工作负载的当前分布情况来分发请求。 如果对未来的推理有利，它还会复制或交换 KVCache 的某些块。

这张图展示了KVCache 块的存储和传输逻辑。存储方面；在 CPU 内存中，KVCache 以分页块的形式存储。 根据请求模式，它可以采用如 LRU（最近最少使用）、LFU（最不经常使用）或基于请求特征的算法等缓存淘汰算法。当然，KVCache也可以存储在SSD上，同样由CPU完成管理。这里只举了内存的例子，应该是因为SSD方案速度太慢，不满时SLO。

GPU如果需要获取KV Cache，就可以通过CPU 和 GPU 之间的一个名为 Messenger 的独立（基于 GPUDirect）RDMA 组件处理。

图中展示一种请求的典型工作流程。一旦 Token 化完成，Conductor 就会选择一对预填充（prefill）节点和一个解码（decoding）节点。然后进行：

  1. KVCache 重用： Conductor 选定的Prefill节点（组）会接收到一个请求，其中包括原始输入、可重用的前缀缓存块 ID，以及分配给该请求的完整缓存块 ID。Prefill节点根据前缀缓存块 ID 将前缀缓存从远程 CPU 内存加载到 GPU 内存中。如果不存在前缀缓存，则跳过此步骤。
  2. 增量预填充（Incremental Prefill）： Prefill节点（组）利用前缀缓存（prefix cache）完成预填充阶段，并将新生成的增量 KVCache 存回 CPU 内存中。 如果未缓存的输入 Token 数量超过特定阈值（prefill_chunk），预填充阶段将被分割为多个分块（chunk），并以流水线方式执行。 选择该阈值旨在充分利用相应 GPU 的计算能力，通常大于 1000 个 Token。
  3. KVCache 传输： 前述 Messenger 服务部署在每个节点中，用于管理和传输这些缓存。 每个 Messenger 在其各自的推理实例中作为独立进程运行，接收信号以促进高速、跨机器的 KVCache 传输。 此步骤采用异步执行，并与上述增量预填充步骤重叠，将每一模型层生成的 KVCache 流式传输到目标解码节点的 CPU 内存，从而减少等待时间。
  4. 解码（Decoding）： 当所有 KVCache 均接收至解码节点的 CPU DRAM 后，该请求将以连续批处理（continuous batching）的方式加入下一批次。 Conductor 根据当前负载预先选择解码节点，以确保其不会违反 TBT SLO。 然而，由于预估负载在预填充阶段后可能发生变化，本地调度程序会对该 SLO 进行二次检查。 这种二次检查可能会导致请求被拒绝，在这种情况下，相应的预填充成本将被浪费。

Prefill Pool的实现

Prefill加速

Mooncake 利用了仅解码器（decoder-only）Transformer 的自回归特性，并为长上下文预填充实现了分块流水线并行（CPP）。 我们将预填充集群中的每 X 个节点分为一个流水线预填充节点组。 对于每个请求，其输入 Token 会被划分为若干分块，每个分块的长度均不超过 prefill_chunk。 同一请求的不同分块可以由不同节点同时处理，从而实现处理并行化并降低 TTFT。

CPP 提供了两大主要优势：1) 与训练中的流水线并行类似，它仅在每个流水线阶段的边界处需要跨节点通信，这可以轻松与计算重叠。 这带来了更好的 MFU，并减少了与 KVCache 传输之间的网络资源争用。 2) 它天然适配短上下文和长上下文，不会给短上下文预填充带来明显的额外开销，也避免了频繁的节点分区动态调整。

显存优化

除了计算能力外，有限的显存（VRAM）大小也是一种宝贵的资源。

从理论上讲，如果一个请求的 KVCache 大小为 S，处理时间为 T，则其占用成本为 S∗T。 如果请求被分块，且每个分块的处理在分块预填充中与其他解码请求交错执行，T 将会增加，从而导致更大的占用成本。

由于预填充是逐层处理的且属于计算密集型任务，因此可以将 KVCache 的传输和卸载与计算过程重叠，从而进一步降低其占用成本。

在 Mooncake 中，KVCache 的加载和存储通过启动（launch）和等待（wait）操作异步执行。 在每一层的注意力计算开始之前，模型会等待该层 KVCache 的异步加载完成，并触发下一层的 KVCache 异步加载。 在注意力计算完成后，系统会启动该层 KVCache 的异步存储。 一旦所有层的计算完成，流程将等待所有异步存储操作完成。

以 KVCache 为中心的调度

背景知识

问题抽象

“模型即服务”（MaaS）提供商，他们需要解决的问题：如何提高LLM服务的工作负载，可以看作是解决一个具有多重复杂约束的优化问题。优化目标是最大化整体有效吞吐量（这直接影响收入），同时约束条件反映了不同层级的 SLO。

什么是SLO

SLO 通常涉及满足延迟相关的需求，主要是首个 Token 延迟（TTFT）和 Token 间延迟（TBT）。

预填充阶段和解码阶段具有不同的特性，MaaS 提供商设定了不同的指标来衡量其相应的服务水平目标（SLO）。 具体而言，预填充阶段主要关注请求到达与生成第一个 Token 之间的延迟，即首字延迟（TTFT）。 另一方面，解码阶段专注于同一请求中连续生成 Token 之间的延迟，即 Token 间延迟（TBT）。

确保满足服务协议中定义的服务水平目标（SLO）指标对于质量保证至关重要。 例如，TTFTP90=4× 这样的指标表示 90% 的推理请求其 TTFT 不会超过在相同条件下无干扰运行的单个请求 TTFT 的四倍。

PD分离

常见的 Transformer 类模型均采用仅解码器 (decoder-only) 结构。 每个推理请求在逻辑上分为两个阶段：预填充 (prefill) 阶段和解码 (decoding) 阶段。

在prefill（预填充）阶段，所有输入 Token 被并行处理。 该阶段会生成第一个输出 Token，同时存储计算得到的键 (key) 和值 (value) 的中间结果，即 KVCache。

这里的并行计算是指，用户输入的所有 Token都是已知的，可以一并进行Attention 计算

decoding（解码）阶段随后利用该 KVCache 进行自回归生成新的 Token，并将计算过程中产生的新 Key 和 Value 添加到 KVCache 中。

prefill和decoding阶段具有不同的计算特点。在prefill阶段，由于能够同时处理输入 Token，该阶段通常具有较高的计算密集度，短请求除外。当用户输入的序列长度为N时，Attention的时间复杂度是O(N² · d)。N 很大时（比如处理长文档），计算量会急剧上升。

相比之下，由于自回归生成的限制，解码阶段在每个批次中一次仅处理一个 Token。 在引入KV Cache技术后，每一轮只需要计算当前token和历史token的attention，他的时间复杂度是O(N · d)。但是解码阶段需要存储用户输入token的KV Cache和新生成token的KV Cache，这使得解码阶段受限于内存，并导致计算时间随批处理大小呈次线性增长。
